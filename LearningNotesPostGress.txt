CHAPTER 3 — Relationships & Foreign Keys (Hands-On)
Goal of this chapter
    -Understand table relationships
    -Create foreign keys
    -Enforce referential integrity
    -See PostgreSQL’s strict behavior (vs MySQL)

    -- Create child table with foreign key
CREATE TABLE child (
    id SERIAL PRIMARY KEY,
    parent_id INT NOT NULL,
    CONSTRAINT fk_parent
        FOREIGN KEY (parent_id)
        REFERENCES parent(id)
);

-- Foreign key options
ON DELETE CASCADE
ON DELETE SET NULL
ON DELETE RESTRICT

-- Drop foreign key
ALTER TABLE child DROP CONSTRAINT fk_parent;

-- Add foreign key
ALTER TABLE child
ADD CONSTRAINT fk_parent
FOREIGN KEY (parent_id)
REFERENCES parent(id);

CHAPTER 4 — JOINS & QUERYING RELATED DATA (Hands-On)
Goal of this chapter
    -Query data from multiple tables
    -Use different JOIN types
    -Understand real-world query patterns
    -Avoid common JOIN mistakes

    -- INNER JOIN
SELECT * FROM a
JOIN b ON a.id = b.a_id;

-- LEFT JOIN
SELECT * FROM a
LEFT JOIN b ON a.id = b.a_id;

-- RIGHT JOIN
SELECT * FROM a
RIGHT JOIN b ON a.id = b.a_id;

-- FULL JOIN
SELECT * FROM a
FULL JOIN b ON a.id = b.a_id;

-- Aggregation with JOIN
SELECT a.col, COUNT(b.id)
FROM a
LEFT JOIN b ON a.id = b.a_id
GROUP BY a.col;

-- HAVING
HAVING COUNT(b.id) > 0;

INNER JOIN  → matches only
LEFT JOIN   → all left + matches
RIGHT JOIN  → all right + matches
FULL JOIN   → everything
CROSS JOIN  → cartesian
NATURAL JOIN → auto (avoid)

LEFT table  = before LEFT JOIN
RIGHT table = after LEFT JOIN

LEFT JOIN  → keep left table rows
RIGHT JOIN → keep right table rows
Swap tables → swap LEFT/RIGHT behavior

Isolation Levels:
----------------
PostgreSQL supports 4 levels:
Level	Description
READ UNCOMMITTED -	Treated as READ COMMITTED
READ COMMITTED - Default
REPEATABLE READ - Stable snapshot
SERIALIZABLE - Highest safety

CHAPTER 8 — Views, Functions & Procedures (Hands-On)
Goal of this chapter
    -Reuse SQL logic
    -Reduce complex queries in app code
    -Move safe logic into DB

FUNCTIONS vs PROCEDURES:
Feature     	    Function	Procedure
Returns value	    Yes	        No
Use in SELECT	    Yes	        No
Transactioncontrol  No	        Yes

When to Use (Real Life)
Use Views when:
Same JOIN used everywhere
Reporting queries
Cleaner app code

Use Functions when:
Reusable read logic
Validation logic
Calculations

Use Procedures when:
Multiple writes
Transaction logic
Business workflows

-- Views
CREATE VIEW v AS SELECT ...;
CREATE OR REPLACE VIEW v AS SELECT ...;
DROP VIEW v;

-- Functions
CREATE FUNCTION f() RETURNS type LANGUAGE sql AS $$ $$;
SELECT f();

-- Procedures
CREATE PROCEDURE p() LANGUAGE plpgsql AS $$ $$;
CALL p();

-- Difference
Function → returns value
Procedure → executes logic
